# План реализации агента для покупки VPS

В данном документе описан поэтапный план разработки агента, подбирающего оптимальный сервер под avito_mom_bot с учетом жестких требований по биллингу (строго до 2000 руб за 3 мес, без годовых депозитов). Агент будет написан на чистом Python с прямыми вызовами LLM API (OpenAI/Anthropic/Gemini) и размещен в отдельном модуле `infra_agent/` (или `agents/`) внутри проекта `avito_mom_bot`, чтобы не смешиваться с aiogram-хэндлерами. Запуск будет осуществляться из отдельного CLI-скрипта или админ-команды.

## Этапы разработки

### Спринт 1: Базовая линейная версия (AI-пайплайн с мок-данными)
**Оценка времени:** 2-3 часа
**Цель:** Собрать скрипт, проходящий детерминированный путь от Шага 0 до Шага 5 (описанный в обновленном `pipeline_design.md`).
1. **Создание структуры и заглушек:**
   - Модуль `avito_mom_bot/agents/vps_agent/`.
   - Создание файлов мок-данных JSON для Selectel и RUVDS.
   - Базовый CLI-скрипт (`run_agent.py`) для вызова пайплайна.
2. **Реализация узлов конвейера:**
   - Интеграция с LLM API (базовый клиент).
   - `Шаг 0`: Логика подтверждения требований к железу и лимита 2000 руб.
   - `search_tariffs` и `analyze_billing` (чтение из моков + фильтрация).
   - `compare_candidates` (внедрение формулы грубого скоринга в коде для всех провайдеров сразу + объяснение выбора от LLM).
   - `generate_checkout_flow` (вывод разметки с вшитой проверкой безопасности "сверься с экраном").
   - `validate_result` (функция, проверяющая фактический тариф и записывающая плохие тарифы в "память").
3. **Промпт-инжиниринг.**
   - Отладка системных промптов на мок-данных.

### Спринт 2: Переход от моков к реальным данным
**Оценка времени:** 2-3 часа
**Цель:** Заменить JSON-заглушки на простые HTTP-запросы к публичным страницам провайдеров.
1. Реализация парсеров для сбора актуальных данных тарифов.
2. Тестирование пайплайна на реальных данных. Убеждаемся, что алгоритмы фильтрации биллинга и скоринг выдерживают непредсказуемость реальных сайтов.

### Спринт 3: Оборачивание в Агентный Цикл
**Оценка времени:** 2–4 часа
**Цель:** Добавить адаптивность (цикл Observe -> Think -> Act -> Check).
1. Уход от линейного пайплайна к самостоятельному выбору тулов агентом (tools API/function calling).
2. Настройка базы памяти (SQLite или JSON-файл) для запоминания "засад" (из validate_result) и использования этого опыта в Шаге 0/1.

## Следующий шаг
Ожидаем подтверждения плана для старта работ по **Спринту 1**. Мы создадим директорию `agents/`, напишем CLI-скрипт инициализации и создадим мок-контент тарифов для тестирования пайплайна.
